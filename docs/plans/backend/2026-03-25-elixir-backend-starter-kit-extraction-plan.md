# Elixir Backend Starter Kit Extraction Plan

Date: 2026-03-25  
Owner lane: backend  
Status: draft ready for execution

## 1) Objective

Create a reusable **starter-kit platform** from the current backend by extracting stable, high-value capabilities into:

1. **Core libraries (Hex/workspace packages)** for runtime building blocks.
2. **Mix generators** for predictable scaffolding and policy-safe defaults.
3. **A template repository** that composes those libraries and generators into a production-ready app baseline.

The target outcome is a repeatable bootstrap path where a new product team can run one command, get a backend with authentication, authorization, GraphQL Relay conventions, analytics instrumentation, background job processing, and secure operational defaults, then customize domain-specific features.

---

## 2) Scope And Non-Goals

### In Scope

- Authentication and session/token lifecycle foundations.
- Authorization and policy enforcement patterns across GraphQL and data access.
- GraphQL schema conventions (Relay-first nodes/connections/pagination/error shapes).
- Background job infrastructure and workflow primitives.
- Analytics event contracts, emission APIs, and delivery adapters.
- Data layer conventions (timestamps, identifiers, migrations, auditing hooks).
- Generator and template UX for creating/expanding projects.
- Documentation and upgrade/migration guidance.

### Out of Scope (first extraction wave)

- Product-specific domain modules (feed ranking logic, unique business workflows).
- Heavy observability platform integrations beyond minimal pluggable adapters.
- Non-Postgres persistence backends.

---

## 3) Extraction Design Principles

1. **Convention over configuration with explicit escape hatches.**
2. **Secure-by-default** behavior for auth, tokens, GraphQL IDs, and authorization.
3. **Relay-first GraphQL** API design (nodes, global IDs, connections, edges, cursors).
4. **Ownership checks at every boundary**: refetchers and child resolvers must re-apply viewer authorization.
5. **Stable interfaces** between libraries; domain app code should depend on behaviors/contracts, not internals.
6. **Incremental extraction**: move code in vertical slices with compatibility shims.
7. **Generator idempotency**: generated code can be re-run safely for additive updates.

## 4) Proposed Starter Kit Package Topology

## 4.1 Core packages

### A. `three_q_core`

Shared foundation:

- App config loader and runtime validation.
- Ecto baseline helpers.
- Common error structs and normalization.
- Tracing/logging metadata helpers.
- Time helpers locked to `:utc_datetime_usec` usage.

### B. `three_q_data`

Data and schema conventions:

- Base schema macro enforcing:
  - `bigint` primary keys by default.
  - `:entropy_id` UUIDv7 generated by Postgres.
  - `timestamps(type: :utc_datetime_usec)`.
- Explicit opt-in schema mode for UUID-only exceptions (e.g., token tables).
- Migration helpers for indexes, constraints, and entropy IDs.
- Auditable pattern helpers (`inserted_by`, `updated_by`, or event hooks where relevant).

### C. `three_q_auth`

Authentication primitives:

- User identity boundaries (email/password and provider-ready hooks).
- Token issuing/verification/revocation APIs.
- Persisted token hashing with SHA3 family defaults.
- Session/account lockout, verification, and password reset flow primitives.
- Cluster-safe rate-limit interfaces for auth entrypoints.

### D. `three_q_authorization`

Authorization and policy composition:

- Policy behaviors (`can?/3`, scope filters, resource guards).
- Ecto query scoping helpers.
- GraphQL resolver guard helpers (viewer/resource checks).
- Node resolver authorization wrappers to prevent IDOR bypass.

### E. `three_q_graphql`

Relay-first GraphQL toolkit:

- Node/global ID helpers and typed decoding.
- Connection/cursor pagination utilities.
- Standard mutation payload and error envelope conventions.
- Resolver context helpers (viewer, request metadata, authz context).
- Macros/patterns for secure field resolvers and child-object checks.

### F. `three_q_jobs`

Background processing abstractions:

- Oban-oriented behavior layer (workers, retries, uniqueness, idempotency keys).
- Workflow primitives (enqueue, deduplicate, schedule, fan-out/fan-in helpers).
- Job telemetry and failure classification.
- Optional adapter seam for non-Oban backends in future.

### G. `three_q_analytics`

Analytics contracts and pipeline adapters:

- Event schema definition DSL (name/version/properties).
- Validation and coercion for event payloads.
- Event emit API with synchronous and async dispatch.
- Delivery adapters (database stream, queue, webhook provider).
- PII tagging/redaction conventions.

### H. `three_q_testing` (optional but recommended)

Shared test harnesses:

- Factory conveniences and fixtures aligned with starter-kit conventions.
- Contract tests for policies, GraphQL nodes, and job workers.
- Generator test helpers for snapshot validation.

## 4.2 Tooling packages

### I. `three_q_gen`

Mix generators:

- `mix three_q.new` – generate a full backend template instance.
- `mix three_q.gen.context` – context + schema + GraphQL type/query/mutation stubs.
- `mix three_q.gen.auth` – install auth flows and token storage.
- `mix three_q.gen.policy` – policy module + resolver guards.
- `mix three_q.gen.job` – worker, enqueue API, and retry policy skeleton.
- `mix three_q.gen.event` – analytics event contract + emitter wrapper.

### J. `three_q_template` (repo template)

A consumable starter repository that references libraries and includes:

- Bootstrapped config for local/dev/test/prod.
- Example vertical slice (Account + Profile + FeedItem-like module).
- CI defaults, formatting/linting/typecheck/test scripts.
- Upgrade playbook and compatibility matrix.

## 5.3 GraphQL conventions

Extract:

- Relay node/global ID helpers.

Standardize:

- Schema macro patterns to enforce Relay-first design.

Deliverables:

- `three_q_graphql` package.

## 5.4 Analytics

Extract:

- Event contract definitions.

Standardize:

Deliverables:

- `three_q_analytics` package.

## 5.5 Background jobs

Extract:

- Worker behavior scaffolds.

Deliverables:

## 5.6 Data conventions

Extract:

- Base schema/migration helpers for IDs, timestamps, and constraints.

Standardize:

- Explicit UUIDv7-only exception path.
- `:utc_datetime_usec` timestamps everywhere.

Deliverables:

- `three_q_data` package.
- Generator defaults aligned with these conventions.

## 6) Generator And Template UX

## 6.1 Project bootstrap (`mix three_q.new`)

Inputs:

- app name, module name, Postgres database name, auth mode, GraphQL mode, jobs on/off.

Outputs:

- New app wired with:
  - Repo + baseline migrations.
  - GraphQL schema with Relay primitives.
  - Auth + policy modules.
  - Oban/job config.
  - Analytics event bus skeleton.
  - CI/test/typecheck scripts.

## 6.2 Additive generators

- `three_q.gen.context`: add bounded context with schema + policy + GraphQL integration.
- `three_q.gen.auth`: add/upgrade auth subsystems in existing apps.
- `three_q.gen.job`: create worker + service-layer enqueue wrapper + tests.
- `three_q.gen.event`: define validated analytics event contract.

## 6.3 Generated code quality bar

- Generated modules include typespecs for public functions.
- Generated resolvers include explicit viewer auth checks.
- Generated tests include positive and forbidden-path coverage.
- Generated docs explain which files are safe to customize.

## 7) Execution Roadmap (Milestone-Oriented)

## Milestone 1 — Domain audit and candidate mapping (1 week)

- Inventory existing modules by concern (auth, authz, gql, jobs, analytics, data).
- Tag each module as:
  - directly extractable,
  - extractable with refactor,
  - app-specific (non-extract).
- Define public API draft for each planned package.

Exit criteria:

- Signed extraction matrix and package boundaries doc.

## Milestone 2 — Core foundations (`three_q_core`, `three_q_data`) (1-2 weeks)

- Build foundational packages first.
- Introduce base schema/migration macros and shared errors/time/config helpers.
- Integrate into current app through compatibility wrappers.

Exit criteria:

- Existing app compiles/tests with core libs integrated.

## Milestone 3 — Auth + Authorization + GraphQL (`three_q_auth`, `three_q_authorization`, `three_q_graphql`) (2-3 weeks)

- Extract runtime APIs and guard helpers.
- Enforce Relay-first and authorization invariants through helper APIs.
- Port one complete vertical slice as proof (query, mutation, node refetch).

Exit criteria:

- Proof slice runs entirely through extracted libs with no security regression.

## Milestone 4 — Jobs + Analytics (`three_q_jobs`, `three_q_analytics`) (1-2 weeks)

- Extract worker and event infrastructures with adapters.
- Add idempotency and event-contract validation defaults.

Exit criteria:

- At least two production flows run through extracted job/event paths.

## Milestone 5 — Generators (`three_q_gen`) and template repo (2 weeks)

- Implement bootstrap and additive generators.
- Build template project and snapshot tests.

Exit criteria:

- `mix three_q.new demo_app` produces a runnable backend with docs/tests passing.

## Milestone 6 — Hardening and first adopter migration (1-2 weeks)

- Use starter kit for one new service or internal spike.
- Record friction points and patch generator ergonomics.
- Publish migration and upgrade guides.

Exit criteria:

- First adopter success report and v1.0 release checklist complete.

## 8) How Extraction Happens (Execution Mechanics)

### 8.1 Repository and ownership model

- Keep LiveCanvas as the proving-ground application during extraction.
- Create a dedicated multi-repo workspace for starter-kit packages:
  - `three_q_core`
  - `three_q_data`
  - `three_q_auth`
  - `three_q_authorization`
  - `three_q_graphql`
  - `three_q_jobs`
  - `three_q_analytics`
  - `three_q_gen`
  - `three_q_template`
- Assign one package owner (or pair) per library to keep API surfaces coherent.
- Maintain one “contract test suite” repo that verifies all packages together against a generated app.

### 8.2 Vertical-slice extraction workflow

For each concern (auth, authz, GraphQL, jobs, analytics):

1. Pick one LiveCanvas flow currently in production-like use.
2. Introduce an adapter boundary in LiveCanvas (facade module + behavior).
3. Move implementation into the corresponding `three_q_*` package.
4. Wire LiveCanvas to the package through the new boundary.
5. Run parity checks (tests, GraphQL behavior, authorization checks, token/job/event flows).
6. Remove obsolete local implementation only after parity is proven.

This keeps extraction incremental and avoids risky “big bang” rewrites.

### 8.3 Compatibility and rollout strategy

- Use feature flags/config toggles for high-risk swaps (auth and authorization internals).
- Support temporary dual-path execution where needed:
  - old LiveCanvas path (fallback),
  - new package path (primary during rollout).
- Add explicit deprecation windows (for example: 2 release cycles) before removing compatibility shims.
- Require changelog + migration notes in each package release PR.

### 8.4 Cross-project distribution model (core business goal)

To ensure fixes/features land in all projects with minimal repeated work:

1. Implement the capability once in the relevant `three_q_*` library.
2. Publish a new package version with migration notes.
3. Upgrade dependency versions in each app (including LiveCanvas and future projects).
4. Run each app’s compatibility test suite.
5. Release apps independently after package-upgrade validation.

This enforces a “build once in shared libraries, consume everywhere” model rather than copy/paste reuse.

## 9) Testing, Quality Gates, And Release Policy

## 9.1 Per-package quality gates

- Unit tests for package APIs.
- Integration tests against a template-generated app.
- Security tests for token handling and authorization bypass scenarios.
- GraphQL contract tests for node IDs and connection pagination.

## 9.2 Repo-level verification commands

Define a single starter-kit verification sequence:

1. `mix format --check-formatted`
2. `mix compile --warnings-as-errors`
3. `mix test`
4. `mix typecheck` (for touched typed code)
5. `mix credo` (if enabled)
6. `mix dialyzer` (where available)
7. Template smoke test: generate app, migrate DB, run tests.

## 9.3 Versioning

- SemVer for each extracted package.
- Compatibility matrix documenting starter template version -> package versions.
- Changelog entries grouped by package and migration impact.

## 10) LiveCanvas Migration Plan (Concrete)

### 10.1 Baseline alignment before first extraction

- Freeze and document current LiveCanvas behavior for:
  - authentication flows,
  - authorization outcomes,
  - GraphQL node/refetch + pagination contracts,
  - key jobs and analytics event emissions.
- Capture these as regression tests (golden-path + forbidden-path cases).

### 10.2 Lane-by-lane migration sequence for LiveCanvas

1. **Foundation first**: adopt `three_q_core` and `three_q_data` primitives in schemas/migrations.
2. **Auth migration**: route token/session flows through `three_q_auth`; keep a fallback path until parity is complete.
3. **Authorization migration**: move policy checks and resolver guards to `three_q_authorization`, prioritizing node fetchers and child resolvers to prevent IDOR gaps.
4. **GraphQL migration**: align Relay helpers and mutation/error conventions via `three_q_graphql`.
5. **Jobs + analytics migration**: swap worker/event plumbing to `three_q_jobs` and `three_q_analytics`.
6. **Generator backfill**: codify all validated patterns in `three_q_gen` and ensure `three_q_template` emits the migrated baseline.

### 10.3 Done criteria for LiveCanvas migration

- LiveCanvas runs exclusively on `three_q_*` paths for in-scope concerns.
- Local compatibility shims are removed or explicitly marked for timed removal.
- LiveCanvas green suite includes package contract tests and security regression checks.
- A “new project bootstrap” dry run succeeds from `mix three_q.new`.

## 11) Ongoing Upgrade Strategy Across All Projects

1. Keep product apps thin: business logic local, platform concerns in `three_q_*` libs.
2. Upgrade libraries on a regular cadence (for example weekly or per release train), not ad hoc.
3. Maintain an app-level compatibility checklist so every project upgrades consistently.
4. Backport only emergency patches; all normal fixes/features land in shared libs first.

Suggested rollout order for existing apps:

1. `three_q_core` / `three_q_data`
2. `three_q_auth`
3. `three_q_authorization`
4. `three_q_graphql`
5. `three_q_jobs`
6. `three_q_analytics`
7. generators + template finalization

---

## 12) Risks And Mitigations

- **Risk: Over-generalization.**  
  Mitigation: extract only repeated, stable patterns; keep app-specific flows local.

- **Risk: Security regressions during extraction.**  
  Mitigation: mandatory auth/authz contract tests and resolver guard helpers.

- **Risk: Generator drift from runtime libs.**  
  Mitigation: snapshot tests + generated-app integration tests in CI.

- **Risk: Adoption friction due to too many options.**  
  Mitigation: opinionated defaults, minimal required inputs, advanced options behind flags.
